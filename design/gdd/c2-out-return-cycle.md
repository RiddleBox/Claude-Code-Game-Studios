# C2 — 外出-归来循环（Out & Return Cycle）

> **Status**: Approved
> **Author**: Claude Code Game Studios
> **Last Updated**: 2026-03-27
> **Implements Pillar**: 真实存在感（角色有自己的生活）/ 陪伴不打扰

## Overview

C2 外出-归来循环是游戏的核心 Idle 机制。它管理角色何时「外出」到异世界、在外多久、以及何时「归来」。系统采用**保底+随机**的触发机制：每次归来后启动最短保底间隔（默认 20 分钟），期间每分钟 tick 积累出发概率，达到阈值后尝试触发外出。若角色正忙（交互中），C2 通过 F2 的延迟出发机制管理重试。外出期间角色处于 Away 状态，C2 通过 Fe2 泄漏内容系统偶尔推送「它正在经历什么」的暗示。归来时触发碎片演出。**外出时角色会携带一个可交互的次元入口（洞口/笔记本等世界观载体）同行，因此玩家仍然可以通过这个入口发出指令；只是外出态的响应表现与窗口内常态互动不同，会以弱响应、延迟响应或任务后回馈的方式呈现。**对于离线期间，若离线时长超过阈值（默认 4 小时），启动时直接触发一次「归来」事件（携带离线期间遭遇），不逐分钟模拟。

## Player Fantasy

> 玩家感受到的不是「宠物不在了」，而是「它有自己的行程」。它会离开窗口，去做自己的事，留下一个可交互的入口让你随时联系它；如果你发出指令，它不会像普通异步任务那样冷冰冰地等待，而是会在外出态里用不同的方式抽空回应你。你会感到它既独立又在意你：它不是永远围着你转，但也不是彻底断开连接。归来时，它带回来的不是数值奖励，而是「你刚才错过了它的一段人生」——这让陪伴感更真实，也让它的世界继续活着。

## Detailed Design

### Core Rules

#### 1. 基本职责
1. C2 是角色“外出-归来”行为的唯一管理者，负责：
   - 触发外出
   - 记录外出开始与结束时间
   - 管理外出期间的泄漏内容节奏
   - 管理归来触发与离线归来
2. C2 不直接控制动画、声音或对话内容；它只向 F2 发送 `departure_triggered` / `return_triggered`，并接收 `departure_accepted` / `departure_declined`
3. C2 依赖 F3 的分钟 tick 进行时间推进，依赖 F4 存档持久化外出状态

#### 2. 外出触发节奏
4. 每次外出结束后，进入**保底冷却期** `MIN_OUTING_INTERVAL`，默认 **20 分钟**
5. 冷却期结束后，C2 每次收到 F3 的 `tick(current_timestamp, delta_minutes)` 时：
   - 累积 `restlessness`
   - 计算是否达到外出尝试阈值
6. 外出尝试阈值采用“保底+随机”：
   - 在冷却期内：绝不尝试外出
   - 冷却期后：每分钟增加概率权重
   - 达到上限时强制尝试出发，避免无限拖延
7. 若 F2 由于当前状态忙碌而返回 `departure_declined`，C2 继续保持压力值，并在后续 tick 继续尝试，直到成功或达到强制出发条件

#### 3. 外出开始与结束
8. 当 C2 决定尝试外出时，向 F2 发送 `departure_triggered`
9. 若 F2 返回 `departure_accepted`，C2 记录：
   - `outbound_start_timestamp`
   - 当前外出状态进入 `Away`
10. 若 F2 返回 `departure_declined`，C2 记录一次失败尝试，不重置外出压力，只延后重试
11. 外出持续时间由 C2 记录，达到预定外出时长后向 F2 发送 `return_triggered`
12. 默认外出时长为**随机区间**（例如 10–45 分钟，具体数值后续在 Tuning Knobs 确定）

#### 4. 外出期间的入口同行
13. 外出后，角色仍携带“次元入口”这一交互载体（洞口 / 笔记本等世界观表现）
14. 玩家仍可通过该入口向 C2 发出指令
15. 外出态下的指令不直接等同于窗口内互动，而是进入“外出响应模式”：
   - **弱响应**：简单确认、文字/特效回馈
   - **延迟响应**：待当前外出流程空档后再处理
   - **任务后回馈**：归来时一起结算并反馈
16. 外出态下，C2 只允许有限度的指令处理；不会中断当前外出状态去立刻返回

#### 5. 归来触发
17. 外出时长到期后，C2 发送 `return_triggered`
18. F2 进入 Returning，完成归来演出后回到 Talking 或 Idle
19. 归来后，C2 触发一次碎片/遭遇结算，将外出期间内容交给 Fe2 / Fe1 / C3（后续依赖）

#### 6. 离线处理
20. 游戏启动时，C2 从 F3/F4 读取离线分钟数与上次外出状态
21. 若离线时长小于阈值：按正常启动流程恢复，不额外触发归来
22. 若离线时长**大于等于 OFFLINE_RETURN_THRESHOLD**（默认 4 小时）：
   - 视为外出在离线期间已经完成
   - 启动后直接进入一次“离线归来结算”
   - 不逐分钟模拟外出过程
23. 离线归来结算会生成外出期间的“结果摘要”，供归来演出使用

#### 7. 与 F2 的协作
24. C2 不强制打断玩家正在进行的交互
25. 当 F2 处于 Interacting / Talking / Reacting(B) / Performing / Returning 时，C2 只累计压力并等待再次尝试
26. 当 F2 处于 Idle / Attentive / Reacting(A) 时，C2 可立即出发

### States and Transitions

C2 维护以下状态：

| 状态 | 描述 |
|------|------|
| **Standby**（候命） | 角色在窗口内，等待下次外出时机；累积 `restlessness` |
| **Away**（外出中） | 角色已外出，正在异世界经历事件；管理泄漏内容节奏 |
| **Returning**（归来中） | 已触发 `return_triggered`，等待 F2 完成归来演出 |
| **OfflineReturn**（离线归来） | 启动时检测到离线超过阈值，直接进行归来结算 |

**状态转换表**：

| 当前状态 | 触发条件 | 目标状态 | 说明 |
|---------|---------|---------|------|
| Standby | `restlessness` 达到阈值，F2 接受出发 | Away | 正常外出触发 |
| Standby | `restlessness` 达到阈值，F2 拒绝出发 | Standby | 保持压力，继续等待 |
| Away | 外出时长到期 | Returning | 触发归来 |
| Away | 玩家通过入口发出指令 | Away | 不切换状态，进入外出响应模式 |
| Returning | F2 完成归来演出 | Standby | 重置 `restlessness`，开始新冷却期 |
| *(启动)* | 离线时长 < 阈值 | Standby | 正常恢复 |
| *(启动)* | 离线时长 ≥ 阈值 | OfflineReturn | 直接进行离线归来结算 |
| OfflineReturn | 结算完成 | Standby | 进入候命，开始新冷却期 |

**状态图**：

```
                    ┌─────────────────────────────────────┐
                    │              Standby                │
                    │  (累积 restlessness, 等待时机)  │
                    └─────────────┬───────────────────────┘
                                  │
              F2 接受 departure   │   F2 拒绝 departure
              ─────────────────►  │  ─────────────────►
                                  │   (保持压力，继续等待)
                                  ▼
                    ┌─────────────────────────────────────┐
         ◄─────────│               Away                  │
         │          │  (外出中，泄漏内容，响应入口指令)     │
         │          └─────────────┬───────────────────────┘
         │                        │
         │              外出时长到期
         │                        ▼
         │          ┌─────────────────────────────────────┐
         │          │            Returning                │
         │          │    (等待 F2 完成归来演出)            │
         │          └─────────────┬───────────────────────┘
         │                        │
         │              演出完成   │
         └────────────────────────┘

    启动时离线 ≥ 阈值：
    ─────────────────────────────────────────► OfflineReturn ──► Standby
```

### Interactions with Other Systems

| 交互系统 | 数据流向 | C2 提供什么 | C2 需要什么 |
|---------|---------|------------|------------|
| **F2 角色状态机** | 双向 | `departure_triggered` / `return_triggered` 信号 | `departure_accepted` / `departure_declined` 信号；F2 当前状态查询（用于判断是否可出发） |
| **F3 时间/节奏系统** | F3 → C2 | — | `tick(current_timestamp, delta_minutes)` 信号，用于累积外出压力和计时 |
| **F4 存档系统** | 双向 | 写入 `outbound_start_timestamp`、`current_c2_state` | 读取上次外出状态、离线时长计算 |
| **Fe2 泄漏内容系统** | C2 → Fe2 | `away_content_tick` 信号（外出期间定期触发泄漏） | — |
| **Fe1 对话系统** | C2 → Fe1 | 归来时的碎片/遭遇数据包 | — |
| **C3 碎片系统** | C2 → C3 | 外出期间生成的碎片内容 | — |
| **P1 主界面 UI** | C2 → P1 | 当前状态（Standby/Away/Returning）供显示 | — |
| **F5 Aria 接口层** | 双向 | 外出态下的指令接收与响应 | 指令内容、响应方式 |

**接口约定**：
- C2 暴露信号 `outbound_triggered()` — 供 Fe2/Fe5 感知外出开始
- C2 暴露信号 `return_completed(fragments: Array)` — 归来时广播碎片数据
- C2 暴露只读属性 `current_state: C2State`、`is_away: bool`
- C2 暴露方法 `handle_away_command(command: String) -> AwayResponse` — 供 F5 在外出态下调用

## Formulas

#### 躁动值积累

每次收到 F3 的 `tick` 时，若当前处于冷却期后（`standby_minutes >= MIN_OUTING_INTERVAL`）：

```
restlessness += RESTLESSNESS_PER_MINUTE

触发尝试条件：
  roll = random_float(0.0, 1.0)
  if roll < restlessness / RESTLESSNESS_MAX:
      尝试出发（向 F2 发送 departure_triggered）

强制出发条件（无需 roll）：
  if restlessness >= RESTLESSNESS_MAX:
      强制尝试出发
```

**变量定义**：

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `MIN_OUTING_INTERVAL` | 归来后最短冷却时间（分钟） | 20 |
| `RESTLESSNESS_PER_MINUTE` | 每分钟累积的躁动值 | 5 |
| `RESTLESSNESS_MAX` | 躁动值上限，达到后强制出发 | 100 |
| `OUTING_DURATION_MIN` | 外出最短时长（分钟） | 10 |
| `OUTING_DURATION_MAX` | 外出最长时长（分钟） | 45 |
| `OFFLINE_RETURN_THRESHOLD` | 触发离线归来的最短离线时长（分钟） | 240（4小时）|

**示例计算**：
- 冷却期结束后第 1 分钟：`restlessness = 5`，触发概率 = 5/100 = **5%**
- 冷却期结束后第 10 分钟：`restlessness = 50`，触发概率 = 50/100 = **50%**
- 冷却期结束后第 20 分钟：`restlessness = 100`，达到上限，**强制出发**
- 最早可能外出时间：冷却期 20 分钟 + 躁动期 0 分钟（极低概率）= **最短 20 分钟**
- 最晚强制出发时间：冷却期 20 分钟 + 躁动期 20 分钟 = **最长 40 分钟**

#### 外出时长随机

```
outing_duration = random_int(OUTING_DURATION_MIN, OUTING_DURATION_MAX)
```

示例：随机范围 10–45 分钟，均匀分布，每次外出长度各不相同。

## Edge Cases

**1. 游戏启动时角色处于 Away 状态（上次退出时正在外出）**
- 条件：F4 存档记录 `current_c2_state = Away`，且离线时长 < `OFFLINE_RETURN_THRESHOLD`
- 处理：恢复 Away 状态，根据剩余外出时长继续计时；若外出时长已超出，立即触发归来

**2. 游戏启动时角色处于 Away 状态，且离线时长 ≥ 阈值**
- 条件：`current_c2_state = Away`，离线时长 ≥ 240 分钟
- 处理：跳过剩余外出计时，直接进入 OfflineReturn，生成离线期间的结果摘要并触发归来演出

**3. F2 持续拒绝出发（玩家长时间处于交互状态）**
- 条件：`departure_declined` 连续多次，`restlessness >= RESTLESSNESS_MAX` 且 F2 仍不接受
- 处理：C2 保持强制出发意图，每 tick 继续尝试；不强制中断玩家交互。若 F2 状态在 `MAX_DEPARTURE_WAIT`（默认 10 分钟）内始终不允许，记录警告日志（异常情况）

**4. 外出期间游戏意外退出（进程崩溃）**
- 条件：Away 状态未正常写入存档就崩溃
- 处理：由 F4 的定时自动保存（每 10 分钟）保证最多丢失 10 分钟的状态；下次启动时按最后存档的状态恢复

**5. Fe2 泄漏内容系统未就绪时收到 `away_content_tick`**
- 条件：C2 尝试推送泄漏内容，但 Fe2 尚未设计/实现
- 处理：MVP 阶段 Fe2 未实现时，`away_content_tick` 信号可以广播但无接收方，C2 不报错，静默丢弃

**6. 外出时长随机结果为最小值（10 分钟）时玩家刚好在交互**
- 条件：外出 10 分钟即到期，但 F2 处于 Talking/Performing 状态
- 处理：`return_triggered` 发出后，F2 根据状态优先级决定何时执行；C2 等待 F2 完成当前状态后归来，不强制打断

**7. 离线归来结算时 C3/Fe1 尚未就绪（MVP 阶段）**
- 条件：`return_completed(fragments)` 广播时，C3 碎片系统尚未实现
- 处理：碎片数据暂存在 C2 内存中，等待 C3 就绪信号后再交付；MVP 阶段可简化为直接丢弃并记录日志

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 增大效果 | 减小效果 |
|------|--------|---------|---------|----------|
| `MIN_OUTING_INTERVAL` | 20 分钟 | 10 到 60 分钟 | 角色更少外出，陪伴时间更长 | 外出更频繁，节奏更活跃 |
| `RESTLESSNESS_PER_MINUTE` | 5 | 2 到 10 | 达到阈值更快，外出窗口更早开启 | 积累更慢，外出时机更随机分散 |
| `RESTLESSNESS_MAX` | 100 | 60 到 200 | 强制出发时间更晚，随机窗口更宽 | 更快强制出发，随机性降低 |
| `OUTING_DURATION_MIN` | 10 分钟 | 5 到 30 分钟 | 每次外出更长，归来更少 | 外出更短，归来更频繁 |
| `OUTING_DURATION_MAX` | 45 分钟 | 20 到 120 分钟 | 外出时长上限更大，偶尔很久不见 | 外出时长上限缩短，节奏更紧凑 |
| `OFFLINE_RETURN_THRESHOLD` | 240 分钟（4小时） | 60 到 480 分钟 | 离线更久才触发归来，适合长时间不用 | 离线较短即触发归来，更敏感 |
| `MAX_DEPARTURE_WAIT` | 10 分钟 | 5 到 30 分钟 | F2 拒绝后等待更久再警告 | 更快记录异常，调试更灵敏 |

**调优说明**：
- 关键体验目标：玩家一个工作日（8小时）内应感受到 **3-6 次**外出-归来循环
- 当前参数下：最短循环约 30 分钟（20 冷却 + 最短触发 + 10 外出），最长约 85 分钟（40 强制出发 + 45 外出）
- 建议先以默认值运行原型，观察实际节奏感后再调整

## Visual/Audio Requirements

#### 视觉需求

C2 本身不渲染任何内容，但它的状态变化驱动以下视觉表现（由其他系统实现）：

| C2 事件 | 视觉表现 | 负责系统 |
|---------|---------|----------|
| Standby → Away（出发） | 角色淡出消失（C1 Away 动画）；窗口内视觉切换为「外出态」 | C1、P1 |
| Away 期间 | 窗口内显示「次元入口」可交互态（替代角色位置） | P1 |
| Away → Returning（归来） | 角色从洞口探入的 `return_in` 动画 | C1 |
| 外出响应模式（Away 时收到指令） | 入口载体产生弱反馈特效（如光晕脉冲、文字浮现） | P1、Fe2 |

**外出态视觉设计（待 P1 设计时确认）**：
- 洞口/窗口框架始终可见，外出不改变窗口结构——类似屏幕黑屏但屏幕还在
- 角色消失后，洞口「内部」变暗或变空，可有轻微环境动画（如黑暗中偶尔的光点、微弱呼吸感）
- 不需要大幅替换 UI：框架不动，只是内部内容消失
- 具体的暗场效果（渐黑方式、粒子等）由美术方向和 P1 设计确定，C2 只定义状态

#### 音频需求

| C2 事件 | 音效建议 | 负责系统 |
|---------|---------|----------|
| 出发时 | 轻柔的「离开」音效（如风声、脚步渐远） | Fe5 |
| 归来时 | 对应的「到来」音效（如碰撞轻响、惊喜音） | Fe5 |

| `is_away: bool` | C2 | P1 切换「常态视图」/「外出态视图」 |
| `current_state: C2State` | C2 | P1 精细控制各子状态的视觉表现 |
| `outbound_triggered()` 信号 | C2 | P1 触发出发动画序列 |
| `return_completed()` 信号 | C2 | P1 触发归来动画序列，显示碎片获得提示 |

**外出态 UI 设计原则**：
- 洞口/窗口框架本身**始终存在**，外出不改变主体画面结构
- 外出期间「里面」变暗/变空——类似屏幕黑屏但屏幕还在，或传送门熄灭但门框还在
- UI 不需要大幅重排：角色区域内容消失，背景/窗框保持，可交互入口保留
- 归来时：角色从洞口内重新出现（亮起），无需重建 UI 结构
- 具体的「内部暗场」效果（渐黑、粒子消散等）由 P1 设计时确认

## Acceptance Criteria

- [ ] 游戏运行 20 分钟后（冷却期结束），角色开始有概率外出；20 分钟内绝不触发外出
- [ ] 冷却期结束后最多 40 分钟内，角色必定完成一次外出（强制出发）
- [ ] 外出触发时，若 F2 处于 Idle/Attentive/Reacting(A)，立即进入 Away 状态
- [ ] 外出触发时，若 F2 处于 Interacting/Talking/Performing，`departure_declined`，C2 保持躁动值继续等待，不重置
- [ ] Away 状态持续到外出时长到期后，C2 发送 `return_triggered`，角色完成归来演出
- [ ] 归来完成后，`restlessness` 重置为 0，冷却期重新开始计时
- [ ] 外出期间，玩家通过入口发出指令，C2 进入外出响应模式，不中断 Away 状态
- [ ] 关闭游戏时 C2 状态写入存档；重启后状态正确恢复
- [ ] 离线时长 < 4 小时重启：恢复上次状态，按剩余时长继续计时
- [ ] 离线时长 ≥ 4 小时重启：直接触发离线归来结算，完成后进入 Standby
- [ ] 外出时长随机分布在 `OUTING_DURATION_MIN`–`OUTING_DURATION_MAX` 区间内，不固定
- [ ] 一个工作日（8 小时）模拟运行中，外出-归来循环次数在 3–6 次之间（可用自动化测试验证）
